## 0F.1 Minimum criterion for valid output

### Purpose
This section defines the minimum threshold that any serious system output must satisfy in order to:
- enter a report,
- sustain interpretation,
- influence prioritization,
- or serve as visible support for assisted decision.

Its function is to prevent outputs that are:
- apparently useful,
- well written,
- visually convincing,
- or operationally convenient

from entering a serious flow without meeting minimum visible admissibility conditions.

This section does not replace other blocks. It functions as the final gate of visible admissibility.

### Cross-cutting rule of governance proportionality
Governance burden must not be uniform for every output.

As a cross-cutting rule of Block 0F:
- `Lane_A`: exploratory / internal / low-impact; requires core gates, clear tagging, and basic traceability;
- `Lane_B`: visible / serious Decision-grade; requires sufficient review, visible limits, and clear dependency;
- `Lane_C`: Verification-grade / high-impact / report material; requires the full standard of visible admissibility.

The hardest gates must concentrate on `Lane_C`, high-weight recommendations, and persuasive artifacts, not on every useful early output by default.

### Formal definition of valid output
A valid output is a system output that, for a delimited use, simultaneously satisfies minimum conditions of:
- epistemological support,
- traceability,
- honest representation,
- explicit dependency,
- visible limits,
- and semantic discipline

sufficient to appear in a serious flow, report, or assisted decision without inducing material overreading.

A valid output is not defined only by apparent usefulness. It must be admissible both in content and in representation.

### Master rule
Nothing enters the serious report only because it sounds good, looks good, or seems useful. Every visible output must be admissible in content and in representation.

Accordingly:
- it is not enough that the base claim exists;
- it is not enough that the output is readable;
- it is not enough that the artifact helps visually;
- and it is not enough that an aggregate score is high.

### Distinction between content and representation
Content validity = whether the claim or output has enough basis to exist for that use.

Representation validity = whether the way it is shown honestly represents that content without inflating it, oversimplifying it badly, or deforming it.

It must remain explicit that:
- an output may be valid in content but invalid in representation;
- and in that case it must not enter in its current form.

Visible admissibility therefore requires double discipline:
- sufficiency of base;
- and honesty of representation.

### Minimum admissibility conditions
Every serious output must satisfy, at minimum:

1. explicit claim linkage;
2. traceable context;
3. delimited use;
4. consistent regime and mode;
5. honest representation;
6. visible limits;
7. sufficient review;
8. absence of open destructive conflict;
9. adequacy to audience and format;
10. possibility of future review.

If one of these conditions materially fails, output admissibility is compromised even if other dimensions score high.

### Rule of explicit dependency
Every valid output must make it possible to answer:
- which claim it depends on;
- which evidence set it depends on;
- which structured analytical output it depends on;
- which restrictions it depends on;
- and what would make it invalid.

Without that explicit dependency, the output may be elegant, but not serious.

### Rule of visible limits
Not every limit needs to be in the main body, but no material output may hide critical limits that change its reading.

Typical critical limits are:
- domain of validity,
- dominant assumptions,
- relevant open conflict,
- exploratory character,
- strong sensitivity,
- non-confirmation,
- partial scope.

A valid output may be brief, but it may not be misleadingly brief.

### Rule of falsifiability or review condition
Every serious output must leave traceable at least one of these two things:

1. what would falsify it;
2. what would force its review.

This is especially important for:
- causal hypotheses,
- projections,
- exploratory claims,
- recommendations,
- and powerful visual artifacts.

An output that leaves no trace of falsifiability nor of review condition tends to become ungoverned rhetorical closure.

### Rule of no overreading
The system must prevent outputs that reasonably invite a stronger interpretation than the one actually supported.

Output validity includes control over how it will probably be read, not only how it was written.

Typical examples:
- table that looks like definitive ranking when it is screening;
- chart that looks like verification when it is projection;
- paragraph that looks like closed causality when it is hypothesis;
- recommendation that looks like mandatory decision when it is preliminary prioritization.

Foreseeable overreading is an admissibility failure, not merely a style risk.

### Minimum rule by object type

#### `Claim-like output`
It must have at minimum:
- claim linkage;
- intended_use;
- minimum basis;
- limits;
- review.

#### `Recommendation-like output`
It must have at minimum:
- objective;
- base claim;
- critical dependency;
- what would change it.

#### `Artifact-like output`
It must have at minimum:
- explicit semantics;
- claim linkage;
- allowed/prohibited interpretation;
- sufficient review.

### Mandatory binary gates
Critical conditions exist such that, if they fail, the output becomes inadmissible regardless of its aggregate score.

At minimum, the framework must evaluate these binary gates:
- `has_claim_linkage`
- `has_traceable_context`
- `has_intended_use`
- `has_required_review`
- `has_visible_critical_limits`
- `has_no_destructive_open_conflict`
- `has_representation_honesty`
- `has_revision_condition`

These gates are of type:
- yes / no
- pass / fail
- mandatory / not mandatory

Their function is to block entry of outputs that fail critical conditions even if they appear strong under scoring.

### Auxiliary scores

#### `content_validity_score`
Measures how strong the content is for the intended use.

#### `representation_integrity_score`
Measures how honestly the representation reflects real support.

#### `traceability_completeness_score`
Measures how reconstructible the output is from claim, evidence, structured analytical output, restrictions, and review.

#### `review_completeness_score`
Measures how complete the review of the output was.

#### `overread_risk_penalty`
Measures how high the risk is that the reader will overinterpret the output.

#### `output_validity_score`
An aggregate score for operational support in ranking, triage, or review prioritization.

These scores do serve for:
- mathematical operations,
- ranking,
- alerts,
- triage,
- and review prioritization.

But they must not replace critical gates nor structural judgment.

### Rule of combination between gates and scores
An output does not enter by average score if it fails a critical binary condition.

Scores help to:
- order,
- prioritize,
- monitor,
- and alert.

Gates decide minimum admissibility.

Accordingly:
- `output_validity_score` does not replace structural judgment;
- an output with high aggregate score may still be inadmissible;
- and there must not be a single magic number that decides everything.

### Summary table

| Minimum condition | What it requires |
|---|---|
| Claim linkage | explicit link to base claim |
| Traceable context | reconstruction from evidence, context, and structured analytical output |
| Delimited use | defined and bounded technical purpose |
| Coherent regime | consistency with Decision-grade or Verification-grade and with output_mode |
| Honest representation | no exaggeration of strength, precision, causality, or closure |
| Visible limits | do not hide domain, critical assumptions, or relevant conflicts |
| Sufficient review | verification mode or equivalent traceable review |
| No destructive conflict | absence of open rupture that destroys admissibility |
| Adequate format | form appropriate to audience and use context |
| Review condition | what would falsify it or what would force review |

### Canonical examples

#### Case 1 — Valid output as a well-limited short preliminary paragraph
“There is sufficient indication to prioritize review of Line A under the observed conditions; this reading remains preliminary and requires additional contrast.”

**Result:** valid if it maintains claim linkage, visible limits, and sufficient review.

#### Case 2 — Invalid output due to misleading representation
An annual projection chart is presented with verificatory-closure aesthetics and a caption suggesting already confirmed result.

**Result:** invalid in representation, even if the underlying content is useful as projection.

#### Case 3 — Invalid output due to incomplete dependency
An executive summary formulates a strong recommendation without declaring base claim, restrictions, or supporting structured analytical output.

**Result:** invalid due to incomplete dependency.

#### Case 4 — Valid output with clear limits
A comparative table shows screening across lines and explicitly states that it does not constitute definitive ranking nor causal confirmation.

**Result:** valid with visible limits.

#### Case 5 — Good aggregate score but failed critical gate
An output obtains good `content_validity_score`, good `representation_integrity_score`, and good `traceability_completeness_score`, but fails `has_visible_critical_limits`.

**Result:** inadmissible. The output does not enter by average score if a critical gate fails.

### Normative closure
The minimum standard of valid output exists to prevent semantically elegant but epistemologically fragile outputs.

The quality of the framework is not measured only by what it can say, but by what it admits itself to show without inducing material overreading.

Therefore, no serious output may enter the visible system flow without simultaneously passing:
- mandatory binary gates,
- auxiliary supporting scores,
- and structural admissibility judgment.

## 0F.2 Minimum standard for valid recommendation

### Purpose
This section defines the minimum standard that a recommendation must satisfy in order to be operationally admissible within the framework.

A recommendation is the point where the system stops only describing, comparing, or hypothesizing, and begins orienting conduct. For that reason it requires a harder standard than a descriptive paragraph or an informative table.

It is also one of the most dangerous forms of applied hallucination if poorly governed, because it transforms technical reading into action suggestion.

### Formal definition of valid recommendation
A valid recommendation is a system output that proposes an action, sequence, prioritization, or next step in an epistemologically admissible way for a delimited use, based on traceable claims, explicit restrictions, visible critical dependencies, and sufficient justification to orient conduct without presenting itself as unquestionable obligation or as a substitute for human decision.

It is not enough that it looks reasonable. It must be reconstructible, reviewable, proportional to support, and semantically honest with respect to the epistemological regime that sustains it.

### Master rule
No recommendation enters the serious system only because it seems reasonable. It must be reconstructible from admissible claims, active restrictions, critical dependency, and review condition.

Accordingly:
- an elegant recommendation without visible base is not a serious recommendation;
- a strong recommendation without compatible base claim is not admissible;
- and a useful but non-reviewable recommendation is not epistemologically governable.

### Distinction between preliminary recommendation and higher-weight recommendation

#### Preliminary recommendation
It orients:
- prioritization,
- measurement,
- review,
- discard,
- analysis sequence,
- and exploratory next step.

It does not claim:
- technical closure,
- strong obligatoriness,
- nor irreversible execution.

Its function is to move investigation, order uncertainty, or define the next useful step under support that is still non-verificatory.

#### Higher-weight recommendation
It may orient:
- more costly technical action,
- strong validation,
- escalation,
- more serious redesign,
- or higher-impact decision.

And it requires:
- more robust base claims,
- greater multilayer convergence,
- lower ambiguity,
- and more explicitly stated critical dependency.

It does not equal automatic executive order. It remains orientation, albeit of higher technical weight.

### Minimum conditions
Every valid recommendation must satisfy, at minimum:

1. explicit base claim;
2. clear objective;
3. visible critical dependency;
4. coherent regime;
5. implicit recognition of cost of error;
6. condition of change or reversal;
7. no false obligatoriness;
8. sufficient review.

If any of these critical conditions materially fails, the recommendation must not be considered valid for serious flow.

### Mandatory critical dependency
Every valid recommendation must include, explicitly or traceably, at least one critical dependency.

Admissible examples:
- depends on the observed thermal load being representative;
- depends on access to measurement under real operating conditions;
- depends on the conflict between historian and invoice being resolved;
- depends on the asset remaining within the evaluated domain.

A recommendation without visible critical dependency tends to be read as absolute advice.

Critical dependency does not weaken the recommendation. It makes it honest and governable.

### Condition of reversal or change
Every valid recommendation must be able to answer at least one of these two questions:

1. What would make this recommendation stop being the best one?
2. What evidence or condition would change the suggested action?

Without condition of reversal or change, the recommendation becomes dangerously similar to non-reviewable mandate.

This is especially serious when:
- the recommendation has high cost,
- the base claim is only Decision-grade,
- or the reader may interpret it as compulsory action.

### Relation with base claim
The strength of a recommendation may never legitimately exceed the strength of the claim on which it depends.

Minimum corollaries:
- an exploratory claim may not sustain strong execution recommendation without mediation;
- a decision-grade claim may sustain preliminary recommendation;
- a verification-grade claim may sustain higher-weight recommendation, but not therefore automatic nor irreversible.

The recommendation inherits:
- limits,
- domain,
- conflict,
- critical assumptions,
- and fragility of the base claim.

If the base claim changes, the recommendation must be reevaluated.

### Relation with Decision-grade / Verification-grade

#### In Decision-grade
Mainly valid recommendations are of the type:
- measure,
- review,
- prioritize,
- discard,
- sequence,
- explore,
- do not close yet.

The main risk is sounding like mandatory decision when only prioritization is being made.

#### In Verification-grade
Higher-weight recommendations may appear:
- delimited redesign,
- validating intervention,
- technical escalation,
- more defensible adjustment,
- recommendation for serious decision.

The main risk is converting strong support into absolute prescription.

Verification-grade authorizes greater technical weight, not prescriptive absolutism.

### Mandatory binary gates
Every recommendation must pass, at minimum, the following binary gates:

- `has_claim_base`
- `has_objective`
- `has_critical_dependency`
- `has_review`
- `has_use_alignment`
- `has_reversal_condition`
- `has_non_overstated_language`

If one critical gate fails, the recommendation must not enter.

These gates are of type:
- yes / no,
- pass / fail,
- mandatory / non-mandatory.

Their function is to block semantically dangerous recommendations even if the aggregate score appears acceptable.

### Auxiliary scores

#### `recommendation_support_score`
Measures how well the recommendation is supported by claims, evidence, structured analytical output, and restrictions.

#### `dependency_clarity_score`
Measures how explicit, visible, and governable its critical dependency is.

#### `actionability_score`
Measures the practical usefulness of the recommendation for prioritizing, sequencing, or activating action.

#### `recommendation_risk_score`
Measures the risk of error or operational overreading if the recommendation is followed or misread.

#### `reversibility_score`
Measures how easily the recommendation may be reviewed, corrected, or reversed if support or context changes.

These scores serve to:
- prioritize review,
- order recommendations,
- detect dangerously weak recommendations,
- and support operational triage.

But they do not substitute binary gates nor structural judgment.

### Rule of severity by cost of error
If the recommendation implies:
- high cost,
- high irreversibility,
- regulatory impact,
- strong financial impact,
- or relevant structural change,

then the following must be required:
- greater support,
- better critical dependency,
- better review,
- and more delimited formulation.

The greater the cost of error, the greater the minimum standard of valid recommendation.

The required severity depends not only on the recommendation as text, but on the potential damage of acting on it.

### Summary table

| Minimum condition | What it requires |
|---|---|
| Explicit base claim | recommendation anchored in one or more traceable claims |
| Clear objective | defined action, sequence, or prioritization |
| Critical dependency | visible condition on which the recommendation depends |
| Coherent regime | consistency with Decision-grade or Verification-grade |
| Sufficient review | verification mode or equivalent review already performed |
| Change condition | what would make it stop being the best option |
| Non-inflated language | does not sound like absolute obligation nor final decision |
| Cost of error considered | severity adjusted to the potential impact of being wrong |

### Canonical examples

#### Case 1 — Valid: preliminary recommendation
“Prioritize point measurement in Line A before proposing redesign, given that the dominant hypothesis still depends on partially observed operation and on contrast not yet completed.”

**Result:** valid as preliminary recommendation if the base claim is Decision-grade and the critical dependency is visible.

#### Case 2 — Valid: delimited higher-weight recommendation
“Under the validated conditions for Line A, escalation to validating intervention on the blowdown scheme is recommended, with scope kept limited to the measured domain and subject to confirmation of operational replicability.”

**Result:** valid as delimited higher-weight recommendation if the base support is sufficiently robust and scope is not universalized.

#### Case 3 — Invalid: inflated recommendation
“The entire system must be replaced immediately.”

**Result:** invalid due to prescriptive absolutism, lack of delimitation, and overextension of the base claim.

#### Case 4 — Invalid: recommendation without critical dependency
“A full operational redesign is recommended.”

**Result:** invalid if it does not declare what it depends on, which conflict must be resolved, or what would make it change.

#### Case 5 — Invalid: recommendation sounding mandatory while exploratory
“This is the correct solution.”

**Result:** invalid due to false obligatoriness, collapse of uncertainty, and absence of semantic discipline.

### Normative closure
A recommendation is not only a useful text. It is an orientation of conduct.

For that reason the framework must demand more from it than elegance:
- base claim,
- critical dependency,
- reversibility,
- and proportionality with the epistemological regime that sustains it.

The quality of the system is at stake not only in what it asserts, but also in how it orients action without turning plausibility into mandate, nor strong support into absolute prescription.

## 0F.3 Minimum standard for valid artifact

### Purpose
This section defines what a table, chart, diagram, annotated image, summarized equation, or visual composition must satisfy in order to appear as a visible and defensible artifact within the framework.

Artifacts have high persuasive power:
- a table may look definitive;
- a chart may look like verification;
- a diagram may look like closed causality;
- a summarized equation may look like strong foundation.

For that reason an artifact requires its own standard.

### Formal definition of valid artifact
A valid artifact is a visual, tabular, symbolic, or multimodal representation of the system that, for a delimited use, translates one or more admissible claims into a form of greater compression or legibility without artificially increasing their epistemological strength, without hiding material limits, and without inducing an interpretation reasonably stronger than the one supported by its base.

It is not enough that it looks clear, fast, or convincing. It must be semantically governable.

### Master rule
No artifact enters the serious report only because it is understood quickly or looks convincing. It must be semantically explicit, epistemologically subordinate to the base claim, and safe against material overreading.

### Distinction between explanatory value and epistemological strength
The artifact may improve:
- understanding,
- informational compression,
- comparative reading,
- executive reading,
- or systemic view.

But it does not by itself increase:
- evidence,
- verification,
- causality,
- nor claim robustness.

An artifact may have high explanatory value and low inherited epistemological strength.

### Minimum conditions
Every valid artifact must satisfy, at minimum:

1. explicit claim linkage;
2. clear artifact purpose;
3. explicit visual semantics;
4. allowed/prohibited interpretation;
5. coherent regime;
6. visible limits;
7. sufficient review;
8. real explanatory value.

### Mandatory visual semantics
Every valid artifact must sufficiently declare what the following mean:
- arrows,
- colors,
- bars,
- nodes,
- lines,
- shadows,
- bands,
- summarized equations,
- groupings,
- or other relevant visual elements.

Minimum examples:
- arrow = flow / sequence / dependency / causal hypothesis / control / correlation / functional relation;
- band = uncertainty / tolerance / expected range / comparison interval;
- color = category / priority / severity / epistemological state.

If the reader may reasonably infer a semantics stronger than the real one, the artifact is not well governed.

### Allowed / prohibited interpretation
Every material artifact must be able to answer:

1. which reading is allowed;
2. which reading is explicitly prohibited.

Admissible examples:
- exploratory structural hypothesis;
- conditioned projection;
- screening table.

Examples of prohibited reading:
- reading a diagram as confirmed causality;
- reading a projection as verified savings;
- reading a table as execution mandate;
- reading a comparison as extrapolable to the whole plant.

### Visible limits
Every valid artifact must make visible, directly or indirectly, limits that materially change its reading, such as:
- domain of validity;
- exploratory character;
- non-confirmation;
- partial scope;
- dominant assumptions;
- conditioned comparison;
- relevant uncertainty;
- overreading risk.

An artifact may be clean, but it may not be silently misleading.

### Relation with base claim
The strength of the artifact may never exceed the strength of the claim on which it depends.

Minimum corollaries:
- exploratory claim → artifact may not look confirmatory;
- decision-grade claim → artifact may not look verification-grade;
- non-confirmation claim → artifact may not be dressed up as near-proof;
- delimited claim → artifact may not universalize the result.

### Mandatory binary gates
Every artifact must pass exactly these minimum gates:

- `has_linked_claim_ids`
- `has_artifact_purpose`
- `has_visual_semantics`
- `has_allowed_interpretation`
- `has_prohibited_interpretation`
- `has_visible_critical_limits`
- `has_required_review`
- `has_regime_alignment`
- `has_real_explanatory_value`

If one critical gate fails, the artifact does not enter.

### Auxiliary scores

#### `artifact_integrity_score`
Measures how integral the artifact is with respect to its base and limits.

#### `visual_semantics_clarity_score`
Measures how clear and non-ambiguous its visual semantics are.

#### `interpretation_control_score`
Measures how well the artifact controls what it allows and forbids the reader to infer.

#### `artifact_explanatory_value_score`
Measures how much real explanatory value the artifact adds for the reader.

#### `artifact_overread_risk_penalty`
Measures the risk that the artifact induces a stronger reading than the one supported.

#### `artifact_validity_score`
An aggregate score for operational support in ranking, alerts, review prioritization, triage, or comparison among artifacts.

#### `semantic_overreach_risk_score`
Measures the specific risk of semantic overextension of the artifact.

#### `visual_density_penalty`
Measures whether visual density creates a false sense of order, closure, or robustness.

These scores serve to:
- order review,
- detect dangerous artifacts,
- prioritize reconstruction,
- or compare alternatives;

but they do not replace binary gates.

### Rule of combination between gates and scores
An artifact does not enter by average score if it fails a critical binary condition.

Scores help to:
- order,
- prioritize,
- monitor,
- and alert.

Gates decide minimum admissibility.

`artifact_validity_score` does not replace structural judgment of the artifact.

### Rule of visual proportionality
Visual density and apparent formal order of the artifact must not create a sensation of greater certainty than the one actually supported.

This applies especially to:
- Sankey,
- causal maps,
- system boundary diagrams,
- priority matrices,
- before/after charts,
- and summarized equations.

The artifact must be visually proportional to the epistemological regime and to the real support of the base claim.

### Summary table

| Minimum condition | What it requires |
|---|---|
| Claim linkage | explicit link to base claims |
| Clear purpose | defined explanatory purpose |
| Visual semantics | sufficient meaning of relevant visual elements |
| Allowed/prohibited interpretation | explicit valid reading and prohibited reading |
| Coherent regime | alignment with Decision-grade or Verification-grade |
| Visible limits | domain, assumptions, uncertainty, non-confirmation, or other relevant limits |
| Sufficient review | traceable semantic review of the artifact |
| Real explanatory value | material explanatory usefulness for the reader |
| No induced overreading | sufficient control of materially excessive reading |

### Canonical examples

#### Case 1 — Well-bounded exploratory artifact
“Exploratory structural hypothesis; arrows indicate plausible dependency, not verified causality.”

**Result:** valid if arrows, domain, and exploratory character are explicitly governed.

#### Case 2 — Valid screening table
“Indicative prioritization; not a definitive ranking nor an execution mandate.”

**Result:** valid if the table declares that it compares preliminarily and does not appear to show unearned closure.

#### Case 3 — Invalid chart due to overextension
A before/after chart presents projected improvement as if it were already verified result.

**Result:** invalid due to overextension.

#### Case 4 — Invalid diagram due to ambiguous semantics
A causal map uses strong arrows without declaring whether they represent sequence, hypothesis, or confirmed causality.

**Result:** invalid due to ambiguous semantics.

#### Case 5 — Pretty artifact with no real explanatory value
A decorative chart repeats exactly what the text already says without adding comparison, structure, or clarity.

**Result:** inadmissible due to lack of real explanatory value.

#### Case 6 — High aggregate score but failed critical gate
An artifact obtains good `artifact_integrity_score`, good `artifact_explanatory_value_score`, and low `visual_density_penalty`, but fails `has_prohibited_interpretation`.

**Result:** inadmissible. An artifact does not enter by average score if a critical gate fails.

### Normative closure
An artifact is not only visual support. It is a surface of interpretation.

For that reason the framework must demand from it:
- explicit semantics,
- subordination to the base claim,
- overreading control,
- and visible limits

before admitting it as serious representation.

## 0F.4 Final admissibility checklist

### Purpose
After all the prior architecture, this block functions as the final control gate.

Without it, each evaluation would again depend on:
- variable human interpretation,
- urgency,
- editorial intuition,
- or contingent operational convenience.

This block exists to decide whether a concrete output may be shown now, in this form, for this use, without inducing material error.

### Formal definition of final admissibility checklist
The final admissibility checklist is a compact structure of validation prior to publication, use, or inclusion in a report.

Its function is to evaluate whether a concrete output satisfies critical minimums of:
- support,
- traceability,
- representation,
- limits,
- review,
- and proportionality.

It does not replace 0F.1, 0F.2, or 0F.3. It compresses them into a final visible and actionable control gate.

### Rule of subordination
0F.4 does not redefine epistemic theory, recommendation validity, or artifact validity. It consumes the primary definitions already fixed in 0F.1, 0F.2, and 0F.3 and produces only a final visible release decision.

### Master rule
Every material output must pass through an explicit final admissibility decision. It is not enough that it exists, is well written, or has a good aggregate score.

The guiding question is only one:
can this output be shown now, in this form, for this use?

### Core checklist

#### A. Claim base and traceability
1. Does it have explicit claim linkage?
2. Does it have sufficient traceable context?
3. Does it have clear or inherited intended_use?

#### B. Epistemological consistency
4. Is it coherent with the applicable epistemological regime?
5. Is it coherent with the lifecycle_status of the base claim?
6. Does it not depend on open destructive conflict?

#### C. Representation and limits
7. Is the representation honest?
8. Are critical limits visible or traceable without material ambiguity?
9. Does probable reading not exceed real support?

#### D. Review and action dependency
10. Did it pass through verification view or sufficient review?
11. Does it have review, change, or falsation condition when applicable?
12. Does its format actually add explanatory value for this context?

### Decision logic

#### Rule 1 — Critical failure
If any of these critical conditions fails, the output is not admissible in its current form:
- claim linkage,
- traceable context,
- coherent regime,
- honest representation,
- sufficient review,
- absence of open destructive conflict.

Typical result may be:
- `Blocked`
- `Rebuild_Required`
- or `Rejected`

#### Rule 2 — Non-critical but material failure
If no critical gate fails, but the following do:
- visible limits,
- overreading control,
- review condition,
- real explanatory value,
- or clarity of critical dependency,

then the output may go to:
- `Approved_with_Warning`
- `Limit_Scope`
- `Rebuild_Required`

depending on severity.

#### Rule 3 — Full approval
The output may remain `Approved` only if:
- no critical gate fails,
- it has no uncorrected material defects,
- and its current form does not induce disproportionate reading.

### Failure severity

#### Type 1 — Structural failures
They affect base admissibility:
- no claim linkage,
- no traceability,
- no review,
- open destructive conflict,
- or strongly misleading representation.

Typical action:
- `Blocked`
- `Rejected`
- or `Rebuild_Required`

#### Type 2 — Scope or reading failures
They affect proportionality:
- domain not explicit,
- inflated caption,
- visually ambiguous semantics,
- recommendation too strong,
- artifact with overreading risk.

Typical action:
- `Limit_Scope`
- `Approved_with_Warning`
- `Rebuild_Required`

#### Type 3 — Optimization or form failures
They do not destroy admissibility, but they affect quality:
- unnecessarily overloaded format,
- redundancy,
- low compression,
- little additional explanatory value.

Typical action:
- `Approved`
- `Approved_with_Warning`
- or future improvement.

### Possible final outcomes
The final checklist may produce exactly one of these outcomes:

- `Approved`
- `Approved_with_Warning`
- `Limit_Scope`
- `Rebuild_Required`
- `Blocked`
- `Rejected`
- `Human_Review_Required`

#### Operative meaning
- `Approved`: the output may be shown in its current form for that use.
- `Approved_with_Warning`: the output may be shown, but with explicit warning.
- `Limit_Scope`: the output may be shown only if its scope or reading is limited.
- `Rebuild_Required`: the output must not be shown like this; it requires new construction.
- `Blocked`: the output may not be used operationally in its current form.
- `Rejected`: the output is inadmissible for that use and in that form.
- `Human_Review_Required`: the output must not circulate without traced human review.

### Rule of human override
Human override must exist, but not as arbitrary erasure of the checklist.

If a human approves something that partially failed, the following must remain traced:
- what failed,
- why it was nevertheless approved,
- for which limited use it is accepted,
- which risk is being assumed.

Human override must not erase the system's epistemological signal. It must coexist with it.

### Rule of compatibility with scores
This checklist may coexist with:
- `output_validity_score`
- `recommendation_support_score`
- `artifact_validity_score`
- and other auxiliary scores.

But it must not be replaced by them.

Scores order; the checklist decides minimum admissibility.

Accordingly:
- a high score does not cancel a critical failure;
- a low score does not necessarily imply automatic rejection;
- and the final decision remains structural, not purely numeric.

### Summary table

| Critical question | If it fails | Typical action |
|---|---|---|
| Does it have claim linkage? | structural failure | `Blocked` or `Rejected` |
| Does it have traceable context? | structural failure | `Blocked` or `Rejected` |
| Is it coherent with regime? | epistemological inconsistency | `Limit_Scope`, `Blocked`, or `Rebuild_Required` |
| Is representation honest? | overassertion or distortion | `Rebuild_Required`, `Blocked`, or `Rejected` |
| Does it have visible limits? | materially risky reading | `Approved_with_Warning` or `Limit_Scope` |
| Did it pass sufficient review? | lack of final control | `Blocked` or `Human_Review_Required` |
| Does it have review condition? | risk of rigid closure | `Approved_with_Warning` or `Limit_Scope` |
| Does it add real explanatory value? | superfluous or noisy output/artifact | `Approved_with_Warning`, `Rebuild_Required`, or do not include |

### Canonical examples

#### Case 1 — `Approved`
Decision-grade paragraph with claim linkage, clear limits, sufficient review, and proportional reading.

**Result:** `Approved`

#### Case 2 — `Approved_with_Warning`
Useful and traceable table, but with domain visible only in a secondary note.

**Result:** `Approved_with_Warning`

#### Case 3 — `Limit_Scope`
Correct verification-grade chart, but the caption suggests extrapolation to the whole plant.

**Result:** `Limit_Scope`

#### Case 4 — `Rebuild_Required`
Useful exploratory diagram, but with ambiguous arrows that look like confirmed causality.

**Result:** `Rebuild_Required`

#### Case 5 — `Blocked`
Recommendation without clear claim base nor sufficient review.

**Result:** `Blocked`

#### Case 6 — `Rejected`
Artifact without clear visual semantics, without linked claims, and not reconstructible from traceable context.

**Result:** `Rejected`

### Normative closure
The final admissibility checklist exists to prevent the system from publishing elegant but epistemologically fragile outputs.

Its function is not to beautify governance, but to make explicit the last decision:
- show,
- limit,
- rebuild,
- block,
- or reject.

No material output must enter serious flow without passing through this explicit final decision.
